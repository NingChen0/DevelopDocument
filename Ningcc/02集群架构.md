  ```shell
  find ${backup_dir} -type f -name "*.tar.gz" -mtime +7 | xargs rm -f
  ```

##### 自动化创建与分发密钥

1. > 阻碍：
   >
   > 1. 分发公钥的时候：yes/no(yes后，会把信息保存到.ssh/known_host)
   > 2. 分发公钥的时候输入密码

2. 自动化创建密钥

   ```shell
   ssh-keygen -f ~/.ssh/id_rsa   -P ''
   # -f：指定了生成的密钥文件的路径和名称
   # -P：密钥的通行短语
   #
   #安装sshpass
   yum install sshpass -y
   #使用sshpass 与ssh-copy-id分发公钥
   先：ssh-copy-id 172.16.1.31  检查是否需要输入yes/no
   然后使用：
   sshpass -p'1234' ssh-copy-id 172.16.1.31
   #sshpass适用于给ssh相关的命令提供密码:ssh,scp,ssh-copy-id.
   #expect是一门语言,用于实现非交互式判断,认证相关的操作.
   
   # 提示yes/no,使用-oStrictHostKeyChecking=no 临时不检查主机信息.
   sshpass -p1234 ssh-copy-id -i ~/.ssh/id_rsa.pub -o StrictHostKeyChecking=no 172.16.1.41
   
   ```

3. 自动化创建与分发脚本

   ```shell
   #!/bin/bash
   #auth :ning
   # date: 10/29
   # this is for assign the ssh-keygen secret key to remote server
   
      #vars
      pass=1234
      ips="192.168.75.101 192.168.75.41 192.168.75.31"
      #create the secret key pair
      if [ -f ~/.ssh/id_rsa ]; then
        echo "已经创建过密钥对"
      else
        echo "正在创建密钥对..."
        ssh-keygen -t rsa -f ~/.ssh/id_rsa -P '' &>/dev/null
      fi
      #assign the public secret key
      for ip in $ips; do
        sshpass -p${pass} ssh-copy-id -i /root/.ssh/id_rsa.pub -o StrictHostKeyChecking=no "${ip}" &>/dev/null
        echo "${ip} 创建成功"
      done
   ```

#### 5.3 .handler触发器

* handler触发器

  > 应用场景: 一般用于配置文件分发时候
  >
  > 如果配置文件发生变化则需要重启服务,如果没有变化则不重启
  >
  > * handler要放在剧本的最后
  >
  > * tasks 和 handlers 部分应该处于同一层级，表示它们都是该Playbook的一部分。
  >
  > * notify 后面跟的是触发器的名字，应当缩进正确，确保它是 copy 模块的一部分。

  * 案例： 使用handler触发器分发配置文件

    ```yaml
    ---
    - hosts: nfs
      tasks:
        - name: 分发配置文件
          copy:
            src: /files/exports
            dest: /etc/exports
            backup: yes
          notify:
            - 重启服务
    
      handlers:
        - name: 重启服务
          systemd:
            name: nfs-server
            state: reloaded
    ```

###### 1. HTTP状态码

| 状态码 | 含义                   |
| ------ | ---------------------- |
| 2xx    | 表示正常               |
| 3xx    | 表示正常，需要进行跳转 |
| 4xx    | 表示异常，客户端问题   |
| 5xx    | 表示异常，服务端问题   |

| 详细的状态码                       | 说明                                                         |
| ---------------------------------- | ------------------------------------------------------------ |
| 200 ok                             | 访问正常                                                     |
| 301 Moved Permanently              | 永久跳转                                                     |
| 302 Found 或 Moved Temporary       | 临时跳转                                                     |
| 304 Not Modified                   | 浏览器缓存，没有更新页面                                     |
| 403 Forbidden                      | 权限拒绝（拒绝访问），权限问题，首页文件问题                 |
| 404 Not Found                      | `文件找不到，一般辅助错误日志进行排查`                       |
| 500 Internal Error                 | 内部错误，SElinux开启，其他原因辅助错误日志排查              |
| 502 bad Geteway                    | 网关错误，一般发生在负载中（负载均衡服务器全挂），请求发送到后面，后面无人处理，提示502 |
| 503 service tempoarily unavailable | 服务临时不可用，后端负载异常等，人为设置（网站升级）         |
| 504 Gateway Time-out               | 网关超时                                                     |

  ```shell
  #作用: 设置传递给后端服务器的X-Real-Ip头部。

  `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  #作用: 设置传递给后端服务器的X-Forwarded-For头部。
  #说明: $proxy_add_x_forwarded_for变量会自动处理现有的X-Forwarded-For头部，并在其基础上添加当前代理服务器的客户端IP地址。如果请求已经经过了多个代理服务器，那么X-Forwarded-For头部会包含一个IP地址列表，每个代理服务器都会在这个列表中添加自己的客户端IP地址。通过这条指令，Nginx会确保后端服务器能够看到完整的客户端IP地址链。
  ```

###### 10.1. 负载均衡轮询算法

* > 决定负载均衡如何把请求分发给后端节点

* 常见的轮询算法；

  * rr，wrr，ip_hash，lc算法（最小连接数），wlc（加权最小连接数）

    | 负载       | 说明                                                         |
    | ---------- | ------------------------------------------------------------ |
    | rr轮询     | round robin轮询，默认的循环访问                              |
    | wrr        | 加权轮询，在轮询基础上加上权重，server中weight就是加权轮询   |
    | ip_hash    | ip哈希，只要客户端IP一样，就会一直访问同一个后端节点（用户请求与服务器绑定）<br />解决会话保持<br />可能导致负载不均 |
    | url_hash   | url_hash只要用户访问的url相同/uri相同，就访问相同的web服务器<br />缓存服务器：静态资源缓存 |
    | least_conn | 最小连接数，lc算法，也可配合上权重weight,wlc权重最小连接数   |

##### 12. 负载均衡和反向代理

* > 区别在于两者处理用户请求的方式
* | 内容     | 共同点                       | 区别                                                         |
  | -------- | ---------------------------- | ------------------------------------------------------------ |
  | 负载均衡 | 将用户的请求分发到后端节点上 | 用户->load balance->web lb负载均衡做的是数据转发，不会产生新的请求，1个请求一个响应 |
  | 反向代理 | 用户的请求分发到后端的节点上 | 中间有一个中介，用户->中介->web 2个请求2个响应，代理代替用户去找web服务器 |

  ```shell

  # pid 文件是运行的nginx的进程号
  ```

* 发送 WINCH 信号
  发送 WINCH 信号给旧的主进程，让其优雅地关闭所有工作进程，并停止接受新的连接。

  ```shell

  ```

* 发送 QUIT 信号：
  最后，发送 QUIT 信号给旧的主进程，让其完全退出。

  ```shell
  kill -QUIT $(cat /var/run/nginx.pid.oldbin)
  ```

   * 使用脚本检查服务是否存在，不存在则关闭keepalived

   ```shell
   	systemctl stop keepalived
   fi
   port_cnt=`ss -lntup |grep nginx |wc -l`
   if [ $port_cnt -eq 0 ];then
      systemctl stop keepalived
   fi
   ```

   ```shell
       state MASTER   #设置主备角色， 备：BACKUP, 单词要大写
   ```

  | 说明                             | tomcat                | nginx                   |
  | -------------------------------- | --------------------- | ----------------------- |
  | 定义访问日志的格式               | Host部分的pattern定义 | http区域log_format 部分 |
  | 客户端IP地址                     | %h                    | $remote_add             |
  | 访问时间                         | %t                    | $local_time             |
  | 请求起始行                       | %r                    | $request                |
  | 状态码                           | %s                    | $status                 |
  | 大小                             | %b                    | $body_bytes_sent        |
  | 从哪里条状过来（如何访问的网站） | %{Referer}i           | $http_referer           |
  | 客户端类型（浏览器）             | %{User-Agent}i        | $http_user_agent        |
  | XFF头记录                        | %{X-Forwarded-For}i   | $http_x-forwarded_for   |

##### 9. Tomcat的ROOT目录

> 在Apache Tomcat服务器中，ROOT目录是指部署在Tomcat服务器上的默认Web应用程序的`根目录`。当用户访问没有指定具体Web应用的URL时（例如，只输入了服务器的IP地址或域名），Tomcat会默认指向这个ROOT Web应用。
>
> `webapps/ROOT`就是`ROOT` Web应用的根目录。如果你将一个名为`index.html`的文件放在`webapps/ROOT`目录下，那么当你在浏览器中访问`http://yourserver:8080/`（假设Tomcat运行在8080端口上）时，Tomcat将会显示这个`index.html`页面。

      ```shell
      #jstack 是一个用于生成Java虚拟机（JVM）的线程转储的工具，它可以帮助开发者了解在某一时刻每个线程的状态和堆栈跟踪
      ```

    线程可能处于以下几种状态之一：

    - **NEW**：线程已被创建但尚未启动。
    - **RUNNABLE**：线程正在JVM内执行，可能正在等待操作系统资源（如处理器）。
    - **BLOCKED**：线程被阻塞，等待获取一个监视器锁以进入同步块/方法。
    - **WAITING**：线程处于无限期等待状态，直到其他线程执行特定的动作。例如，调用了 `Object.wait()`、`Thread.join()` 或 `LockSupport.park()` 方法。
    - **TIMED_WAITING**：线程处于有时间限制的等待状态。例如，调用了 `Thread.sleep()`、`Object.wait(long)`、`Thread.join(long)` 或 `LockSupport.parkNanos()` 等方法。
    - **TERMINATED**：线程已经退出，完成了执行或由于异常而终止

二进制安装方法

```shell

mkdir -p /app/tools/     # 存储软件包
```

2. 设置配置文件

   ```shell
   cat>/etc/my.cnf <<'EOF'
   [mysqld]
   ## 用户
   user=mysql
   ## 安装目录
   basedir=/app/tools/mysql80/
   ## 数据目录
   datadir= /app/data/3306
   port=3306
   socket=/tmp/mysql.sock
   [client]
   socket=/tmp/mysql.sock
   EOF
   ```

3. 修改配置和数据目录所有者

   ```shell
   chown mysql.mysql /etc/my.cnf

   ```

   ```shell
   echo 'export PATH=/app/tools/mysql80/bin:$PATH' >> /etc/profile
   # 
   source /etc/profile
   # 检查是否配置成功
   mysql -V
   ```

5. 初始化数据库

   ```shell
   mysqld --initialize-insecure --user=mysql \
          --basedir=/app/tools/mysql80/ \
          --datadir=/app/data/3306/
   # 检查是否初始化成功
   echo $？
   ```

6. 启动

   ```shell
   #  拷贝已经准备好的启动管理文件
   cp /app/tools/mysql80/support-files/mysql.server /etc/init.d/mysqld

   ```

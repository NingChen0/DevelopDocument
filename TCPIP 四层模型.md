### TCP/IP 四层模型

数据从你的应用程序发出，经过这四层打包，发送到网络，对方再一层层拆包

| 层级 | 名称       | 核心功能                              | 通俗比喻（物流系统）                                         | 代表协议             |
| ---- | ---------- | ------------------------------------- | ------------------------------------------------------------ | -------------------- |
| 4    | 应用层     | 定义数据格式，提供用户接口            | 写信内容：你写的是情书还是账单？用中文还是英文？             | HTTP, DNS, SSH, SMTP |
| 3    | 传输层     | 负责端到端的连接，确保数据完整        | 选择快递服务：是发“挂号信”（保丢必赔）还是“普通明信片”（丢了不管）？ | TCP, UDP             |

### 第 4 层：应用层 (Application Layer) 协议

#### **1. HTTP/HTTPS**协议

(超文本传输协议)：浏览器和服务器对话的语言（“我要看这个网页”）

HTTP 报文结构--请求报文 (Request)

1. **起始行 (Start Line)**：概括核心意图。
2. **头部 (Headers)**：附加信息（像信封上的备注）。
3. **空行 (Empty Line)**：分隔头部和身体。
4. **主体 (Body)**：实际传输的数据（可选）。

```http
GET /index.html HTTP/1.1          <-- 1. 起始行：方法 + 路径 + 版本
Host: www.example.com             <-- 2. 头部：告诉服务器我要访问哪个域名
User-Agent: Mozilla/5.0...        <--    头部：我是谁（浏览器类型）
Accept: text/html                 <--    头部：我能看懂什么格式
Cookie: session_id=abc123         <--    头部：我的身份凭证（登录状态）
                                  <-- 3. 空行：必须有！表示头部结束
(这里没有主体，因为 GET 请求通常不需要发数据)
```

- 请求方法 (Methods)：
  - `GET`：**获取**资源（如看网页、看图片）。参数通常拼在 URL 后面。
  - `POST`：**提交**数据（如登录、发帖）。数据放在 Body 里，更安全，无长度限制。
  - `PUT`：**更新**资源。
  - `DELETE`：**删除**资源。

实际http请求：

```http
POST /User/GetMailInfo HTTP/1.1           # HTTP 1.1 协议POST 请求报文,请求路径/User/GetMailInfo
Accept: application/json, text/plain, */* # “服务器你返回数据时，优先给我 JSON 格式，其次是纯文本，实在不行啥格式都能接受”
Accept-Encoding: gzip, deflate			 #“服务器你返回数据时，可以用 gzip/deflate 压缩，我能解压缩，省流量” 
Accept-Language: CN						#“服务器你尽量返回中文内容”
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9  #身份认证令牌（Token），相当于你的 “通行证”。Bearer是固定格式
Cache-Control: no-cache					#“服务器你别用缓存的数据糊弄我，必须给我最新的”要求不缓存
Connection: keep-alive					#“请求完别断连接，后面可能还要继续请求，省得重新连”
Content-Length: 0					    #“我这次请求的「请求体」（要提交的数据）长度是 0，也就是没带任何数据”
Content-Type: application/x-www-form-urlencoded #“如果我带了请求体，数据格式是表单格式（虽然这次长度 0，没带）”
Cookie: sidebarStatus=0					#客户端的 Cookie 信息，相当于服务器存在你这边的 “小纸条”，这里记录了 “侧边栏状态是 0（比如收起）”
Host: 192.168.100.33:28123				#服务器的地址和端口
Origin: http://192.168.100.33:28123		 #“我是从这个地址的页面发起的请求”（用于跨域验证）
Pragma: no-cache					   #和Cache-Control: no-cache作用一样，是老版本 HTTP 的兼容写法，也是要求不缓存
Referer: http://192.168.100.33:28123/EndPoint/EndPointList/3 #“我是从这个页面跳过来发起请求的”（服务器可以用这个判断请求来源）
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/145.0.0.0 Safari/537.36 Edg/145.0.0.0 			#“我是 Windows10 系统的 Chrome/Edge 浏览器”，服务器可以通过这个识别客户端类型，适配不同的返回格式
```

**HTTP 报文结构--相应报文 (Response)**

返回给浏览器的数据包长这样：

```http

<html>
  <body><h1>Hello World!</h1></body>
</html>
```


*注意：HTTP 本身不负责传输，它依赖底层的 TCP 协议 保证数据可靠到达，所以 HTTP 请求前，必须先进行 TCP 三次握手*

#### **2. http状态码 (Status Codes)**

---- 这是排查问题的关键！

* 2xx (成功)：
   `200 OK`：一切正常，请求成功。
* 3xx (重定向)：
   `301 Moved Permanently`：网址变了，永久跳转（如 http 跳 https）。
   `302 Found`：临时跳转。
* 4xx (客户端错误)：你搞错了。
   `400 Bad Request`：你发的请求格式不对。
   `401 Unauthorized`：没登录，需要认证。
   `403 Forbidden`：登录了，但没权限看这个。
   `404 Not Found`：最经典，资源不存在（网址输错了或文件被删了）。
* 5xx (服务器错误)：服务器搞挂了。
   `500 Internal Server Error`：服务器代码崩了（Bug）。
   `501 Bad Gateway`：网关错误（通常是后端服务挂了，Nginx 连不上）。
   `503 Service Unavailable`：服务器忙不过来（超载）或维护中。

#### **3. DNS**：域名解析 

​	---电话簿。把 `www.google.com` 翻译成 IP 地址（如 `142.250.x.x`）

* 通常基于 UDP 端口 53

* DNS 的两种查询方式

  * **递归查询**

    客户端发起请求后，本地DNS查不到IP，本地DNS服务器必须代劳到底， 本地服务器自己去完成后续（可能用迭代的方式访问根故服务器、顶级域名服务器等）的所有查询，把最终的IP地址返回给客户端

  * **迭代查询**

    客户端发起请求后，本地服务器如未查到，去访问根服务器，跟服务器没有查到，会返回另一个可能知道的DNS服务器地址。

    本地DNS服务器与其他DNS服务器之间使用迭代查询

  客户端浏览器请求访问流程

* DNS缓存

  DNS返回正确的IP之后，系统会把这个结果临时存储起来。并设定一个失效时间，在这个失效时间之内再次访问这个网站时，系统会直接从电脑本地的DNS缓存中把结果交还给你，而不必再次去查询DNS服务器了

* 浏览器DNS查找顺序

  浏览器DNS缓存-->本地系统DNS缓存--> 本地计算hosts文件--> ISP DNS 缓存--> DNS服务器递归or 迭代查询

#### **4. DHCP** 原理

客户端和DHCP服务器四步骤交互（DORA）

> Discovery-- Offer-- request -- acknowledgment

1. **Discover**

   客户端开机连接网络发现本机没有ip

   客户端广播出去一个DHCP Discover 报文（目标mac :`ff:ff:ff:ff:ff:ff`）对应IP `255.255.255.255`

   源IP地址`0.0.0.0 `(尚未拥有IP)

2. **offer **

   一个或多个DHCP收到 客户端发送的Discover后，从地址池中预留一个可用IP

   DHCP 服务器向客户端单播或者广播发送DHCP offer报文（提供的IP，子网掩码，租期，网关和DNS）

   客户端只接受第一个DHCP Offer报文

3. **Request**

   客户端收到第一个DHCP offer 时，，广播一个DHCP Request 报文， 声明这个客户端要用这个IP了。通知其他的DHCP 服务器不用他们提供的IP，请释放这个预留IP

4. **DHCP Acknowledge( 确认)**

   被选中的DHCP 服务器接收到Request 后，正式确认分配，发送DHCP ACK 报文

   客户端收到ACK后，正式使用该IP地址，并开始计时租期

   如果服务器无法分配，则发送DHCP NAK（否定应答）， 客户端重启发送Discover请求

### 第 3 层：传输层 (Transport Layer)协议

​	**职责**：负责**端到端**（进程到进程）的通信。通过**端口号**区分不同的应用程序。保证数据可靠、完整、按顺序、不丢失、不重复地从一端传到另一端

#### 1. TCP 协议

​	**最核心特点**（必须记住）

* **面向连接**：必须先 “建立连接”，才能发数据。→ 像打电话，先拨通，再说话。

* **可靠传输**：丢了会重发、乱了会重排、错了会丢弃。每个数据块都有**编号**，没收到就重发

* **全双工**：双方可以同时收发。

* **字节流**：数据像水流一样连续传输，不分边界。

* **有拥塞控制、流量控制**：不会一股脑乱发，会看网络堵不堵，接收方告诉发送方：你慢点发，我处理不过来。网络堵了 → **自动降低发送速度**，不搞崩网络

**TCP 数据包结构**:

如果把 TCP 数据包比作一张快递单 + 包裹：

1. `源/目的端口` = 寄件人房号 / 收件人房号。
2. `序列号` = 包裹编号（第几箱）。
3. `确认号` = 回执单（“上一箱收到了，请发下一箱”）。
4. `标志位 (SYN/FIN/ACK)` = 特殊指令贴纸（“这是第一箱”、“这是最后一箱”、“请确认”）。
5. `窗口大小` = 备注（“我家门口堆不下了，下次少送点”）。
6. `校验和 `= 防伪封条（破了就不要了）。
7. `数据 `= 箱子里的实际货物。

#### **2. UDP协议**

##### 🚀 一、UDP 的核心特点
* `无连接` (Connectionless)：发送数据前不需要建立连接（没有三次握手）。想发就发。
* `不可靠 `(Unreliable)：
  不保证到达：数据包可能在中途丢失。
  不保证顺序：后发的包可能先到。
  不重传：丢了就是丢了，协议层不管。
* `面向报文 `(Message-Oriented)：UDP 保留消息边界。你发一个 100 字节的包，对方就收到一个 100 字节的包，不会像 TCP 那样把字节流打散或合并。
* `支持一对一、一对多、多对多`：
  天然支持广播 (Broadcast) 和 组播 (Multicast)。TCP 只能一对一。
* `头部开销极小`：只有 8 字节（TCP 最少 20 字节），传输效率更高。

**TCP 与 UDP** 

| 特性          | TCP (传输控制协议)           | UDP (用户数据报协议)       |
| ------------- | ---------------------------- | -------------------------- |
| 连接性        | 面向连接 (需握手)            | 无连接 (直接发)            |
| 可靠性        | 高 (不丢、不乱、重传)        | 低 (可能丢、乱序、不重传)  |
| 速度          | 慢 (头部大，机制多)          | 快 (头部小，无负担)        |
| 传输单位      | 字节流 (无边界)              | 数据报 (有边界)            |
| 通信模式      | 仅点对点 (一对一)            | 一对一、一对多 (广播/组播) |
| 流量/拥塞控制 | 有 (自动减速)                | 无 (应用层自己控制)        |
| 典型应用      | 网页、邮件、文件传输、数据库 | 视频直播、游戏、语音、DNS  |



### 第 2 层：网络层 (Internet Layer)协议

**职责**：负责逻辑寻址（IP地址）、路由选择（决定路径）和分组转发。核心目标是`“把数据包从源主机送到目标主机”`

#### 1. **IP协议**

​	**IP 只做两件事**：

* **写地址**：在包头上写上“从哪里来”（源 IP）和“到哪里去”（目的 IP）。

* **指路**：告诉沿途的每一个中转站（路由器），这个包下一站该往哪送。

> **关键点**：IP 协议**不管**包裹里装的是什么，也**不管**包裹会不会丢。它只管**“尽力而为” (Best Effort)** 地把包往目的地送。如果路断了，它就丢弃，不会像 TCP 那样负责重发。

##### 🚚 **IP 的工作流程**

这是理解 IP 最关键的部分。假设你要从 北京 (电脑 A) 发送一个 IP 包给 纽约 (电脑 B)。

* `步骤 1`：判断目的地在哪？

  电脑 A 拿到目的 IP (纽约)，先拿子网掩码算一下：

  “纽约”和我在同一个局域网吗？

  如果在：直接通过 ARP 协议问“谁是纽约？”，然后发给它。

  如果不在（绝大多数情况）：电脑 A 知道自已送不到，于是把包扔给默认网关（也就是你家的路由器）。

* `步骤 2`：路由器的“查表转发” (核心！)

  路由器收到包后，不做任何感情交流，只做机械动作：

  看目的 IP：哦，要去纽约。 

  查路由表 (Routing Table)：这是一张地图。

  条目 1：去 192.168.x.x -> 走接口 LAN (本地)。

  条目 2：去 10.x.x.x -> 走接口 WAN1。

  条目 3：默认路由 (0.0.0.0) -> 走接口 WAN2 (通往上级运营商)。

  决策：路由器发现纽约不在直连网络里，于是匹配到默认路由。

  转发：把包从 WAN2 接口扔出去，传给下一个路由器（运营商的核心节点）。

* `步骤 3`：逐跳传递 (Hop-by-Hop)

  包到了运营商路由器 -> 查表 -> 扔给省级骨干网路由器。

  到了骨干网 -> 查表 -> 扔给国际出口路由器。

  ... 经过十几个路由器 ...

  最后到了纽约当地的 ISP 路由器 -> 查表 -> 发现目的 IP 就在直连的小区里 -> 最终投递给电脑 B。

  > 💡 关键概念：`TTL (Time To Live) 存活时间`
  >
  > 为了防止包在网络里迷路死循环（比如路由器 A 说去 B，B 说去 A），IP 包头里有个 TTL 字段。
  >
  > 每经过一个路由器，TTL 减 1。（windows  TTL 默认是128，linu 默认是64）
  >
  > 当 TTL = 0 时，路由器直接丢弃该包，并给发送者发个错误报告。
  >
  > 这就是为什么 ping 命令能看到经过了多少个节点。

##### 🛠️ 五、IP 协议的局限性（它不负责什么？）

理解 IP 不做什么，比理解它做什么更重要：

* `不保证送达`：

  路堵了？丢包。

  路由器坏了？丢包。

  IP 耸耸肩：“我尽力了，剩下的事你（TCP）自己解决。”

* `不保证顺序`：

  包 A 可能走高速公路，包 B 走小路。包 B 可能先到。

  IP 不管排序，这也是为什么需要 TCP 来重组顺序。

* `不保证内容完整`：

  虽然 IP 头部有校验和（IPv4），但它只校验头部有没有错。数据部分错了它不管（交给上层 TCP/UDP 校验）。

#### 2. **ARP 协议**

> ARP 的作用：解决“已知目标 IP 地址，求目标 MAC 地址”的问题。

ARP 只在局域网内有效。路由器不会把 ARP 请求转发到互联网上。如果你要访问百度，你的电脑只会 ARP 请求网关（路由器）的 MAC 地址，而不是百度的 MAC 地址。

##### ARP 工作流程

1. **主机 A (`192.168.1.10`)** 想给 **主机 B (`192.168.1.20`)** 发数据，但 A 不知道 B 的 MAC 地址

   * #### 查缓存 (ARP Cache)

     是否有主机B对应的MAC地址表，有则直接发送，没有则下一步启动 ARP 请求流程

   * #### ARP 请求 (ARP Request) —— “广播喊话”

     主机 A 构造一个 **ARP 请求包**

     **目标 MAC** 填全 `F` (广播)，意味着局域网内**所有设备**都会收到这个包，目标 IP 填要找的人 (192.168.1.20)。

   * #### ARP 响应 (ARP Reply) —— “单播回复”

     局域网内所有设备都收到了广播，但只有 **主机 B** 发现：“咦？他在叫我 (`192.168.1.20`)！”
     其他设备默默丢弃该包。

     主机 B 构造一个 **ARP 响应包**，内容是：

     ​	“嘿，主机 A (`AA:AA...`)，我是 `192.168.1.20`。
     ​	我的 MAC 地址是 `BB:BB:BB:BB:BB:BB`。

     此时，主机 B 也顺便把主机 A 的 IP-MAC 对应关系存进自己的缓存里（互利互惠）。

2. ARP 核心逻辑

   | 特性      | 描述                                                      |
   | --------- | --------------------------------------------------------- |
   | 作用      | IP 地址 ➡️ MAC 地址 的映射解析。                           |
   | 工作范围  | 仅限局域网 (二层网络)。跨网段通信时，解析的是网关的 MAC。 |
   | 工作机制  | 广播请求 (Who has?) + 单播响应 (I am here)。              |
   | 缓存      | 有 ARP 缓存表，减少广播风暴，提高效率。                   |
   | 安全性    | 无认证，易受 ARP 欺骗/中间人攻击。                        |
   | IPv6 对应 | NDP (Neighbor Discovery Protocol)。                       |

### 综合案例：当你输入 `www.baidu.com` 并回车时，发生了什么？

为了串联所有协议，我们走一遍全流程：

* `应用层 (DNS)`：浏览器不知道百度的 IP，先问 DNS 协议：“http://www.baidu.com 的 IP 是多少？” -> 得到 110.242.68.3。
* `应用层 (HTTP)`：浏览器构建一个 HTTP 请求：“我要首页”。
* `传输层 (TCP)`：因为 HTTP 需要可靠，所以发起 TCP 三次握手，建立连接。指定目标端口 80。
* `网络层 (IP)`：给数据包加上 IP 头，源 IP 是你的电脑，目标 IP 是百度的 110.242.68.3。路由器根据 IP 协议 决定下一跳去哪。
* `网络接口层 (ARP)`：在你的局域网内，电脑不知道网关（路由器）的 MAC 地址，于是广播 ARP 请求 获取网关 MAC。
* `网络接口层 (Ethernet)`：将数据封装成 以太网帧，通过网线/Wi-Fi 变成电信号发出去。
* `沿途路由`：数据包经过无数个路由器（IP 层转发），最终到达百度服务器。
* `百度服务器解包`：从物理层一直解包到应用层，百度服务器看到 HTTP 请求。
* `返回响应`：百度服务器生成网页 HTML 数据，按原路（反向封装）返回给你。
* `浏览器渲染`：你的电脑收到数据，解包，浏览器把 HTML 渲染成你看到的百度首页。

这就是 TCP/IP 协议族协同工作的完整过程
